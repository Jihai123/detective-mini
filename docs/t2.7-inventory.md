## 3.1 当前 case 数据加载链路

### 调用链（全部静态 import）

```
src/cases/case-001/data.ts
  export const case001Config: StageCaseConfig     ← 数据源，TS 对象字面量

src/cases/case-001/index.ts
  import { case001Config } from './data'
  export const case001: CaseDefinition = { meta, config: case001Config }

src/cases/index.ts
  import { case001 } from './case-001'
  export const CASE_REGISTRY: CaseDefinition[] = [case001, case002]
  export function getCaseById(caseId): CaseDefinition

src/stage1/caseLoader.ts
  import { getCaseById } from '../cases'
  export function loadCaseConfig(caseId: string): StageCaseConfig
    → getCaseById(caseId).config!
    → validateCaseConfig(config)   ← asserts input is StageCaseConfig
    → return config

src/stage1/app.ts (StageOneApp 构造器 / preloadCriticalAssets)
  import { loadCaseConfig } from './caseLoader'
  this.caseConfig = loadCaseConfig(this.state.caseId)   ← 同步调用
```

### 关键特征

- **完全静态 import**：编译期绑定，build 时打包进 JS bundle
- **完全同步**：`loadCaseConfig` 返回 `StageCaseConfig`，无 Promise
- **TypeScript narrowing**：`validateCaseConfig(input: unknown): asserts input is StageCaseConfig`
  — 运行时形状检查（必填字段 + confrontation 双格式），类型断言后直接使用
- **caseLoader.ts API**：
  - 入参：`caseId: string`
  - 出参：`StageCaseConfig`（同步）
  - 副作用：校验失败时抛出 `Error`

## 3.4 类型系统约束

### JSON.parse 后的类型安全性分析

| 类型 | JSON 化后状态 | 风险 |
|------|-------------|------|
| `string` / `number` / `boolean` 标量 | ✅ 无损 | 无 |
| `string[]` / `Record<string, T>` | ✅ 无损 | 无 |
| `Emotion`（string literal union） | ✅ 值保留，TS 类型丢失 | 低：runtime 只读字符串值 |
| `ClueRole`（string literal union） | ✅ 值保留，TS 类型丢失 | 低 |
| `ConditionExpr`（discriminated union by shape） | ✅ JSON round-trip 安全，形状保留 | 低 |
| `HotspotEffect / DialogueEffect`（discriminated union by `type`） | ✅ `type` 字段为字符串，shape 保留 | 低 |
| **`ConfrontationConfig` 双格式** | ⚠️ 需重点关注 | 中 |

### ConfrontationConfig 双格式兼容风险

case-001 data.ts 中同时存在：
- `confrontation.rounds: ConfrontationRound[]`（旧格式）
- `confrontation.suspects: SuspectConfig[]`（新格式 T2.6，镜像数据）

`caseLoader.ts:validateCaseConfig` 已处理此场景：
```ts
const hasLegacyRounds = Array.isArray(c.confrontation.rounds);
const hasMultiSuspect = Array.isArray(c.confrontation.suspects) && c.confrontation.suspects.length > 0;
if (!hasLegacyRounds && !hasMultiSuspect) throw new Error(...);
```

JSON 化后两个字段均保留 → 验证函数通过，**无需额外处理**。

### runtime 是否需要 schema 验证

- 现有 `validateCaseConfig` 是轻量结构检查（asserts 断言），已覆盖关键字段存在性
- JSON.parse 返回 `unknown`，cast 前需调用此函数
- 决策 B 已预设"候选 2 轻量 TS 验证函数，不引入 zod" — 现有 `validateCaseConfig` 可直接复用

## 3.5 Vite 处理 JSON 的方式

### Vite 默认行为（无 vite.config.ts）

Vite 内置 JSON 支持，无需插件：

**方式 1：静态 import（build 时打包进 bundle）**
```ts
import caseData from '/path/to/case-001/data.json';
// → JSON 对象，类型为 any（可 cast 为 StageCaseConfig）
// → build 时内联进 JS chunk，无额外网络请求
```
- Bundle 影响：JS 增加 ~4.8 kB gzip
- 加载延迟：0（随主包加载）
- 适合条件：JSON < 8 kB gzip ✅（case-001 满足）

**方式 2：动态 fetch（运行时单独请求）**
```ts
const data = await fetch('/assets/cases/case-001/data.json').then(r => r.json());
```
- Bundle 影响：JS 不增长，JSON 在 dist 中单独 emit
- 加载延迟：1 次额外 HTTP 请求（~10-50ms，可并行）
- 需要：async loadCaseConfig + loading UI + abort controller

**方式 3：Vite 动态 import JSON**
```ts
const { default: data } = await import('/assets/cases/case-001/data.json');
// → Vite 将 JSON emit 为单独 chunk，异步加载
```
- 与方式 2 效果类似，但走 Vite chunk 系统

### bundle 影响对比

| 方式 | JS 增量（gzip） | 额外请求 | loadCaseConfig 签名变化 |
|------|---------------|---------|----------------------|
| 静态 import | +~4.8 kB/case | 无 | 保持同步 |
| 动态 fetch | 0/case（loader +~0.5 kB） | 1次/case | 变 async（破坏性） |

**条件判断**：case-001 JSON ~4.8 kB gzip < 8 kB → **满足"候选 1 静态 import"条件分支**

## 盘点结论摘要（供决策参考）

1. **加载链路完全同步静态**，改为动态加载会是破坏性变更（需修改 app.ts 初始化流程）
2. **JSON 体积 ~4.8 kB gzip**，满足静态 import 条件（< 8 kB）
3. **数据路径规则统一**（`/assets/cases/{id}/{category}/{file}`），命名约定已在 case-selector 中隐式存在
4. **app.ts 含大量 case-001 硬编码路径**（preload + audio + fallback），这些路径不在 data.ts 中，T2.7-B 迁移 data.ts 后仍需单独处理（或接受为已知遗留）
5. **ConfrontationConfig 双格式**已由现有 validator 兼容处理，JSON 化无风险
6. **现有 `validateCaseConfig`** 可直接作为 JSON cast 后的验证函数，无需引入新验证层
