## 5. endingMatrix 设计

```
rules: [
  // 完美变体(指认陈昊 + 关键证据全 canonical)
  { when: {
      submissionCorrect: true,
      clueInterpretations: { c010: 'canonical' }
    },
    endingKey: 'perfect-B' },

  { when: {
      submissionCorrect: true,
      clueInterpretations: { c010: 'misread' }
    },
    endingKey: 'perfect-C' },

  { when: { submissionCorrect: true },
    endingKey: 'perfect-A' },

  // 错误变体
  { when: { submissionWrongTarget: 'wangkai' },
    endingKey: 'wrong-A' },
]
fallback: 'wrong-B'
```

注:perfect-A vs B vs C 的"总分阈值 14"和"3 条关键证据全 canonical"
判定不在 endingMatrix 表达,而是由 resolveEnding 算法**先判总分和关键证据**,
不达标时降级为"部分结局"(单独 ending key:'partial')。

完整 endingMatrix:
- perfect-A / perfect-B / perfect-C(完美三变体,需总分≥14 + 3 关键 canonical)
- partial(指认对但分数 8-13 或关键证据非全 canonical)
- wrong-A(指认王凯)
- wrong-B(其他错误,fallback)

## 10. 风险点

### 高风险
- attacksTestimonyIds 字段在 evidence.md 中没有,Code 必须按第 4 节表填,
  填错会导致 confrontation 无法击破特定 sentence
- endingMatrix.when 字段名 clueInterpretations / submissionWrongTarget
  是新引入,需要 types.ts 和 resolveEnding 算法同步,任一不匹配直接编译失败

### 中风险
- evidence.md 部分 [DRAFT] 文案区分度偏小(C002/C004 partial 与 canonical
  重叠),进 JSON 后实测可能体感不强 — 接受,T2.8 不修文案,实测后视情况回炉
- 完美/部分/错误三档总分阈值首次启用,实测发现阈值不合理需调参
- C010 触发位置(陈昊工位)需要 scenes 数据中有"陈昊工位" scene,该 scene
  在 testimony.md 中是否独立有定义需在实施时确认 — 若没有,就近放在"办公区"

### 低风险
- ending.md 部分 [DRAFT] 文本可能在玩家实际看到时觉得"过长" —
  接受,实测后视情况调整
- bundle 体积超预算 — 静态 import 体积可控,case-002 data.json 单文件
  即使 12 kB gzip 也在预算内
